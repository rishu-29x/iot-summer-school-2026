# Q20. (Theory) Explain the setup() and loop() functions in Arduino. What happens if you put a long delay() inside loop()? How does this affect sensor reading responsiveness? What is the non-blocking alternative? (2 marks)

In every Arduino program, two functions are mandatory: setup() and loop().

### 1. setup() Function

The setup() function is executed only once when the Arduino is powered on or reset. It is used to initialize the hardware and configure the system before the main program starts.

Typical tasks performed in setup() include:

Setting pin modes using pinMode()
Starting serial communication using Serial.begin()
Initializing sensors, displays, or communication modules

Example:

void setup() {
  pinMode(13, OUTPUT);
  Serial.begin(9600);
}

### 2. loop() Function

The loop() function is executed continuously after setup() finishes. It contains the main logic of the program and keeps repeating until the Arduino is turned off or reset.

Typical tasks include:

Reading sensor values
Controlling LEDs or motors
Sending or receiving data
Making decisions based on inputs

Example:

void loop() {
  digitalWrite(13, HIGH);
  delay(1000);
  digitalWrite(13, LOW);
  delay(1000);
}

In this example, the LED keeps blinking every second because loop() runs repeatedly.

### Effect of a Long delay() Inside loop()

The delay() function pauses the execution of the program for the specified amount of time. During this period, the Arduino does not execute any other code.

For example:

void loop() {
  delay(5000);
  int sensor = analogRead(A0);
}

Here, the Arduino waits for 5 seconds before reading the sensor. During these 5 seconds:

No sensor readings are taken.
No buttons are checked.
No communication takes place.
The program becomes unresponsive.

This is known as blocking execution because the processor is blocked from performing other tasks.

### Effect on Sensor Reading Responsiveness

A long delay() reduces the responsiveness of the system because sensor values are updated only after the delay finishes.

For example, if a motion sensor detects movement during a 5-second delay, the Arduino will not notice it until the delay ends. This can cause:

Missed sensor events
Slow reaction time
Poor performance in real-time applications

In IoT systems, delayed responses may result in incorrect monitoring or control.

### Non-Blocking Alternative

The preferred alternative is to use the millis() function.

millis() returns the number of milliseconds since the Arduino started running. Instead of stopping the program, it checks whether enough time has passed, allowing the Arduino to continue executing other tasks.

Example:

unsigned long previousTime = 0;
const long interval = 1000;

void loop() {
  unsigned long currentTime = millis();

  if (currentTime - previousTime >= interval) {
    previousTime = currentTime;
    digitalWrite(13, !digitalRead(13));
  }

  int sensor = analogRead(A0);
}

### In this program:

The LED blinks every second.
The sensor is read continuously.
The Arduino remains responsive because there is no blocking delay.
Practical IoT Example

Consider a Smart Home Security System with a PIR motion sensor.

Using delay(5000): If motion occurs during the 5-second delay, the Arduino may miss it or respond very late.
Using millis(): The Arduino continuously monitors the PIR sensor while also controlling alarms, LEDs, or sending notifications over Wi-Fi, ensuring immediate response.
